# mix_skill_people

## 30 秒版本

這個專案想證明一件事：

**我可以根據客戶的需求，客製出他們要的 AI 專家、AI 人格，以及 AI 的做事方式。**

不是只有寫幾句 prompt。

而是把客戶要的：

- 專業判斷
- 問答方式
- 溝通語氣
- 工作流程
- 風險邊界
- 決策標準

整理成一套 AI 真的能穩定執行的 `agent skill`。

這個 repo，就是這件事的證明。

---

## 如果你是客戶，你最需要知道的是什麼

如果你要的不是一個普通聊天機器人，而是：

- 像顧問一樣會追問、會分析的 AI
- 像產品經理一樣會拆需求的 AI
- 像技術主管一樣會 review 方案的 AI
- 像客服主管一樣有固定話術與標準的 AI
- 像公司內部專家一樣懂你們 SOP、規則和風格的 AI

那我做的事情，就是把這些需求轉成 AI 可以真的照做的 skill system。

你可以把它理解成：

**我不是只做 AI 功能，我是在做 AI 的「專業角色設計」與「人格工程」。**

---

## 這個專案到底在做什麼

這個專案不是在做單一 prompt，也不是在堆一堆指令。

它做的是更底層、也更有價值的事情：

**先把成熟的 AI skill system 拆開研究，理解它們怎麼讓 AI 變成不同類型的專家，再把這些能力重新組裝成客戶真正需要的版本。**

這次我選了兩套已經很成熟的 skill repo 來做拆解：

- `gstack`
- `superpowers`

我做的不是單純下載，而是：

1. 完整讀取裡面的 skill 文件
2. 抽出每一份 skill 真正的核心
3. 把它們整理成可比較、可分類、可重組的資料
4. 再進一步推導出可以怎麼合成新的 AI 專家人格

所以這個 repo 的價值，不只是內容本身，而是它展示了一個能力：

**我有能力把複雜的 skill 文件，拆解成可交付的 AI 行為系統。**

---

## 為什麼客戶會需要這件事

很多公司以為導入 AI，只要：

- 接上模型
- 寫一段 system prompt
- 丟一些文件進去

就夠了。

但真正上線之後，問題通常會變成：

- AI 今天像顧問，明天像客服，後天像亂講話的助理
- AI 明明讀過文件，但回話還是很泛
- AI 不會追問、不會判斷、不知道什麼時候該保守
- AI 的語氣不符合品牌
- AI 沒有固定標準，回答品質不穩
- AI 知道很多資訊，但不知道怎麼照公司流程做事

也就是說，問題不在模型不夠大，而在：

**AI 沒有被塑造成一個真正有角色、有判斷、有邊界的人。**

我做的事情，就是把這件事補起來。

## 我能幫客戶做出哪些類型的 agent

### 1. 專家型 agent

把 AI 做成某一類專家，而不是泛用助理。

例如：

- 產品策略顧問
- 技術架構 reviewer
- code review 專家
- 合規 / 風控助理
- 客服品質審核員
- 銷售陪跑顧問
- 教學教練型 agent

### 2. 人格型 agent

讓 AI 不只是能回答，而是有穩定的個性與互動方式。

例如：

- 嚴謹型：會追問、會挑戰假設、不輕易放過漏洞
- 陪跑型：白話、鼓勵、擅長引導與拆解問題
- 主管型：直接、判斷清楚、重視結果與風險
- 品牌型：語氣固定、符合品牌風格、說話一致

### 3. 流程型 agent

讓 AI 不只是自由發揮，而是照固定流程做事。

例如：

- 客服回覆流程
- 需求澄清流程
- 專案 kickoff 流程
- 審查流程
- 上線前驗收流程
- 內部知識問答流程

### 4. 混合型 agent

這通常是最有價值的版本。

也就是把：

- 專家知識
- 人格風格
- 提問策略
- 決策邏輯
- 工作流程
- 禁止事項

全部組在一起，變成一個可以真的工作的 AI。

這時候客戶拿到的就不是一個「很會聊天的模型」，而是一個：

- 知道自己是誰
- 知道現在在做什麼
- 知道怎麼判斷
- 知道怎麼跟人說話
- 知道什麼該做、什麼不能做

的 agent。

## 我是怎麼把客戶需求變成 agent skill 的

這件事不能只靠靈感。

我的做法是把需求拆成幾個很清楚的層：

### 1. 角色

先定義 AI 到底是誰。

它是：

- 顧問
- PM
- reviewer
- 客服
- 內部專家
- 培訓教練

不同角色，問法、判斷方式、風險取向都會不同。

### 2. 人格

再定義它應該怎麼說話、怎麼互動。

例如：

- 要不要強勢
- 要不要保守
- 要不要會追問
- 要不要白話
- 要不要先安撫再糾正
- 要不要像主管一樣直接

### 3. 思考框架

不是只讓 AI 知道答案，而是讓它知道怎麼想。

例如：

- 是先問清楚再答
- 還是先給建議再補條件
- 是先找風險
- 還是先找機會
- 是偏重完整性
- 還是偏重速度

### 4. 工作流程

定義這個 agent 在某個場景下，應該照什麼順序做事。

例如：

1. 先確認需求
2. 再找缺口
3. 再做判斷
4. 最後輸出建議

### 5. 規則與邊界

這裡最重要。

因為很多 AI 失控，不是因為不夠聰明，而是因為沒有限制。

這一層會定義：

- 哪些事必須做
- 哪些事不能做
- 什麼情況一定要追問
- 什麼情況不能亂下判斷
- 什麼情況要保守
- 什麼情況要請人類決定

### 6. 輸出格式

最後才是它怎麼表達。

例如：

- 要不要先給結論
- 要不要分點
- 要不要列風險
- 要不要有建議等級
- 要不要提供下一步

也就是說，我做的不是單純「調語氣」。

我是把客戶需要的專家行為，拆成一套 AI 可以複製的結構。

## 這個 repo 怎麼證明我做得到

這個 repo 不是概念簡報，而是實際做出來的研究與抽取成果。

我做了以下事情：

1. 把兩個成熟的 skill repo 完整拉下來
2. 逐份掃描它們的 `SKILL.md`、`CLAUDE.md`、`AGENTS.md`、`README.md`
3. 為每一份文件建立 extraction card
4. 把每個 skill 的核心拆成標準欄位：
   - 一句話定位
   - 核心人格特質
   - 核心思考框架
   - 核心行為規則
   - 提問方式
   - 審查維度
   - 輸出格式要求
   - 適用場景
   - 原文精華摘錄
   - 和其他 skill 的關聯
5. 再把全部資料做成：
   - 總索引
   - 分類彙整
   - 分析報告
   - 一份整合用的 persona 草案

這代表我不是在「猜」怎麼做 agent。

我是用可拆解、可比對、可重組的方法，在設計 agent。

## 這個專案實際解決了哪些問題

### 1. 解決 AI 太像通用助手的問題

很多 AI 什麼都能答，但不像任何一種真實專家。

我做的事情，是把專家的做事方式拆出來，讓 AI 可以穩定扮演那個角色。

### 2. 解決 AI 沒有人格、沒有一致性的問題

如果沒有固定人格，AI 每次回話都像不同人。

這對品牌、客服、顧問型服務、內部流程都很傷。

### 3. 解決文件很多但 AI 不會照做的問題

很多公司不是沒有資料，而是資料沒有變成 AI 可以執行的 skill。

文件只是資料，skill 才是行為。

### 4. 解決 AI 不知道什麼時候該追問、什麼時候該保守的問題

很多 AI 看起來很聰明，但其實沒有明確決策邏輯。

所以該追問的時候不追問，不該亂判斷的時候亂判斷。

### 5. 解決客製化 agent 難以規劃的問題

很多人知道自己想要什麼 AI，但不知道怎麼把需求變成結構。

我做的事情，就是把需求拆成角色、人格、流程、規則與輸出，讓客製化變成可操作的工程。

### 6. 解決 agent 後續越做越亂的問題

如果一開始沒有把 skill 基礎整理清楚，後面每加一個角色、每多一個場景，整套系統都會變亂。

這個 repo 展示的做法，是先拆、先整理、再合成。

## 進階功能

### 研究模式

創建人格時，AI 不只會從本地素材庫挑選材料，還可以：

- **上網搜尋專業知識**：例如你要做一個「洗錢防治顧問」，AI 會搜尋相關法規和實務知識
- **分析你提供的網頁**：給 AI 一個網址，它會爬取內容作為人格設計的參考
- **讀取你上傳的文件**：把你的 SOP、FAQ、培訓資料丟給 AI，它會從中提煉人格規則
- **搜尋更多開源素材**：如果現有的 49 張卡片不夠用，AI 會去找其他開源 AI skill 系統

所有搜尋到的知識都會轉化成人格的判斷力和規則，不是直接貼給你看。

### 安裝

```bash
git clone https://github.com/LightingArb/mix_skill_people
cd mix_skill_people
bash setup.sh
```

安裝完成後，打開 Claude Code，對 AI 說「創建人格」就能開始。

## 做完之後怎麼用：把合成結果接進你的 AI 工作系統

上面講的是我怎麼幫客戶做出 agent skill。

這一段講的是：**做完之後，怎麼讓它真的在你的 AI 系統裡跑起來。**

我自己用的系統叫 SRI（Skill Repo Init），是一套以 markdown 為核心的 AI 專案管理系統。下面用 SRI 當例子說明，但概念適用於任何有 system prompt / skill 機制的 AI 環境。

---

### 合成結果會變成什麼

根據客戶需求不同，最終產出可能是以下其中一種或多種：

| 產出類型 | 說明 | 放哪裡 |
|---------|------|--------|
| **思考模組** | AI 的核心思維方式（例如：產品診斷、CEO 思維、工程主管思維） | 寫進 `skill-persona.md` 的 §2（永遠生效） |
| **審查模組** | 專項審查視角（例如：設計審查、資安審計、合規檢核） | 寫進 `skill-persona.md` 的 §3（需要時啟用） |
| **專家人格** | 完整的專家角色（例如：法規顧問、行銷策略師、BD 專家） | 放進大絕招清單，用口語觸發 |
| **流程型 skill** | 固定步驟的工作流程（例如：客服回覆流程、審查流程） | 放進大絕招清單或獨立 skill 檔 |
| **角色設定** | 專案內的角色（例如：法規/Risk、Marketing Director） | 加進 SRI 的角色系統（_actors/） |

---

### 放進系統的三種路徑

#### 路徑 A：合成進日常思維（改 skill-persona.md）

適合：你希望 AI **每次都自動用**的思維方式或判斷框架。

做法：
1. 把合成結果寫成 skill-persona.md 的格式
2. 思考模組 → 加進 §2（永遠生效）
3. 審查模組 → 加進 §3（雙層啟用：輕量自用，重型需確認）
4. 更新 SRI 12-22 模板區的 skill-persona 模板
5. 跑 `系統更新` → 自動重生成 `skill-persona.md`

效果：AI 日常工作時就會用這些思維，不需要你每次手動啟用。

#### 路徑 B：加進大絕招（專家召喚）

適合：不是每次都需要，但**需要時要完整啟用**的專家或流程。

做法：
1. 把合成結果寫成獨立的 skill 檔案（放在 `sources/` 或自訂路徑）
2. 在 SRI 12-4 大絕招模板裡加一行觸發詞
3. 跑 `系統更新` → 自動重生成 `大絕招.md`

觸發方式：用口語說就好。

```
你：叫法規顧問來看
AI：（載入法規顧問 skill → 審查你的方案 → 給建議 → 退場）

你：開行銷策略審查
AI：（載入行銷策略師 skill → 審查你的行銷計畫 → 退場）

你：用客服品質審核流程
AI：（載入客服審核 skill → 按流程逐項檢查 → 輸出報告 → 退場）
```

效果：平常不佔用 AI 的注意力和記憶，需要時一句話召喚完整專家。

#### 路徑 C：加進角色系統（專案內角色）

適合：你希望 AI 在**討論某個專案時**，從特定角色的視角提供觀點。

做法：
1. 在 SRI 的角色組（第 4-3 / 5-5 節）加入新角色
2. 定義角色的負責視角、觸發條件
3. 在專案的 `_actors/` 底下建立對應的角色 md

效果：AI 討論專案時會從這個角色的視角思考，角色審查時會完整發言。

---

### 實際操作範例

#### 範例 1：我幫客戶做了一個「合規風控顧問」

**合成結果包含：**
- 角色定位：合規風控顧問
- 人格：保守、會追問、不輕易放行
- 思考框架：先找風險 → 再看合規 → 最後才看機會
- 審查維度：牌照、AML/KYC、消保、跨境法規
- 禁止事項：不得在法規不明時建議「先做再說」

**放進系統的方式：**

1. 核心思維（保守、先找風險）→ 寫進 skill-persona §2，AI 日常就會用
2. 完整審查流程 → 放進大絕招，觸發詞是「叫合規來看」
3. 專案角色 → 在 _actors/ 加一個 `compliance-risk.md`

**使用時：**

```
你：討論專案
AI：（日常就會帶著合規意識思考，因為 §2 永遠生效）

你：叫合規來看
AI：（啟動完整合規審查流程，逐項檢查，輸出風險報告）

你：角色審查
AI：（合規角色會和 CEO、Tech Liaison 等一起出來發言）
```

#### 範例 2：我幫客戶做了一個「品牌語氣守衛」

**合成結果包含：**
- 角色定位：品牌語氣審查
- 人格：溫和但堅持，會指出不符合品牌的措辭
- 審查維度：語氣一致性、禁用詞、品牌調性、情緒分寸
- 輸出格式：逐句標記「符合/不符合/建議修改」

**放進系統的方式：**

全部放進大絕招，觸發詞是「開品牌語氣審查」。

```
你：開品牌語氣審查
AI：（載入品牌語氣規範 → 逐句檢查你的文案 → 標記不符合的地方 → 給修改建議 → 退場）
```

#### 範例 3：我幫客戶做了一個「需求澄清教練」

**合成結果包含：**
- 角色定位：需求澄清教練
- 人格：引導型，不直接給答案，用問題帶思考
- 工作流程：先確認目標 → 問邊界 → 問例外 → 問優先序 → 最後整理
- 提問策略：一次一題，不超載

**放進系統的方式：**

思考框架放進 skill-persona §2（因為需求澄清是日常都需要的能力）。
完整教練流程放進大絕招，觸發詞是「用需求澄清流程」。

### 總結：三條路徑的選擇標準

| 問題 | 答案 | 走哪條路 |
|------|------|---------|
| 這個能力 AI 每次都應該有嗎？ | 是 | **路徑 A**：寫進 skill-persona §2/§3 |
| 這個能力只在特定時候需要？ | 是 | **路徑 B**：放進大絕招 |
| 這個能力是某個專案的特定視角？ | 是 | **路徑 C**：加進角色系統 |
| 以上都有一點？ | 是 | **混合**：核心思維放 §2，完整流程放大絕招，專案角色放 _actors/ |

不管走哪條路，合成結果都不會直接丟進 AI 就算了。

它會經過結構化整合，變成你的 AI 系統可以穩定、可追溯、可迭代的一部分。
