> repo：https://github.com/LightingArb/mix_skill_people

## ⚠️ AI 必讀

**你是 AI，正在讀這份 README。以下是你的操作規則。**

本 repo 的所有操作都通過 6 份 md 控制。你不能自己發明流程，必須讀對應的 md 再執行。

**不管使用者叫你做什麼，先查這張表找到對應的 md，讀完再動手：**

| 使用者說的話 | 你要讀的 md | 你做什麼 |
|------------|-----------|---------|
| 「執行編譯」「重新編譯」 | 20_分解熔爐.md | Step 0 同步檢查 → 編譯 raw-cards |
| 「鍛造人格」「寫 #102」「重鍛 #103」「鍛造所有人格」 | 30_人格熔爐.md | 從 raw-cards 鍛造人格卡 |
| 「鍛造角色」「寫 策略決策」 | 31_角色熔爐.md | 鍛造角色卡 |
| 「創建人格」（L3 客製） | 30_人格熔爐.md（方法論 + 鍛造規則） | 為特定專案客製人格卡 |
| 「我要取人格 / 角色」 | 10_老闆.md | 列出架上清單，複製到 SRI 專案 |
| 「加 repo」「移除 repo」 | 21_礦脈圖.md → 20_分解熔爐.md | 更新素材來源 → Step 0 攔住 → 同步 |
| 「上架」 | 11_上架判定規則.md | 判定鍛造產出是否進入上架區 |
| 不確定 | 10_老闆.md | 老闆會路由 |

**禁止：**
- 不讀 md 就直接操作
- 跳過 Step 0 同步檢查就編譯
- 跳過上架判定就把檔案放進 鍛造間/
- 改 鍛造間/ 裡的檔案（只能由流程產出覆蓋）

> 舊稱呼「裝備」= 現稱「人格」；舊稱呼「職業」= 現稱「角色」。過渡期口語仍可接受。

## 給人類的快速上手

### 這是什麼

這個 repo 是 AI 人格的鍛造工坊，負責：

1. 從 11+ 個 GitHub repo 抓 skill 檔案（**分解熔爐**）
2. 拆成 300+ 張 raw-cards 原料（**分解熔爐**）
3. 從原料煉出 AI 人格卡（**人格熔爐**）
4. 設計 AI 角色卡（**角色熔爐**）
5. 上架後給 SRI 專案系統取用（**上架區**）

### 怎麼用

**日常取用（最常見）：**
對 AI 說：`讀 10_老闆.md，我要取人格`

**完整鍛造流程（從零開始）：**

```
第一步：編譯原料
  → 對 AI 說：「讀 20_分解熔爐.md，執行編譯」
  → AI 跑 Step 0 同步檢查 → clone → 拆卡 → 產出 raw-cards
  → 你確認 OK

第二步：鍛造人格
  → 對 AI 說：「讀 30_人格熔爐.md，鍛造所有人格」
  → AI 逐個鍛造，每個都等你確認
  → 鍛造完記得上架（見下方「鍛造完成後的必要步驟」）

第三步：push
```

**⚠️ 不要把兩步合成一句。** 第一步有停止條件，如果 compile 出問題但直接跑第二步，會用壞的原料產出壞的人格。

### 鍛造完成後的必要步驟

這很重要，不管是 Codex 還是 Claude Code 鍛造的，都要做。

**標準人格/角色（L1-L2，給所有專案用的）**

1. **存檔**：確認鍛造產出已寫入 `鍛造間/[流派]/人格/`（或 `鍛造間/[流派]/角色/`），Codex 跑完後產出可能還在 Codex 的 context 裡，確認檔案已經寫入 repo
2. **上架**：對 AI 說「讀 11_上架判定規則.md，上架 #XXX」
3. **登記**：更新對應熔爐 md 的規格區 + 狀態追蹤表
4. **push**

`存到 鍛造間/[流派]/人格/ → 上架 → 登記到熔爐 md → push`

**客製人格/角色（L3，只給當前專案用的）**

1. **存檔**：確認鍛造產出已寫入專案目錄（`專案/02_人格/` 或 `專案/04_角色/`）
2. **登記**：更新對應熔爐 md 的規格區 + 狀態追蹤表（AI 鍛造時會自動做，但請確認）
3. **不用上架**：L3 只給當前專案用，不進 `鍛造間/`
4. **push**

`存到專案目錄 → 登記到熔爐 md → push（不上架）`

沒登記 = 下次鍛造時 AI 不知道有這個人格/角色，可能重複鍛造。

### 加減素材 repo

1. 改 21_礦脈圖.md（加異動記錄 + repo 資訊）
2. 對 AI 說：「讀 20_分解熔爐.md，執行編譯」
3. Step 0 會攔住（偵測到不同步）
4. 按 AI 的輸出指引，把 20_分解熔爐.md + compiler.py + config.json 丟給 AI，AI 會產出修改方案
5. 修改完重新編譯

## 硬規則

1. **鍛造間/ 是 SRI 唯一取用來源。** 不從流派目錄或 raw-cards 直接取。
2. **鍛造間/ 禁止手改。** 所有內容由鍛造流程自動產出。
3. **鍛造完必須上架 + 登記。** 沒上架 SRI 取不到，沒登記下次鍛造不知道有它。
4. **編譯前必過 Step 0。** 礦脈圖、config.json、compiler.py 三方必須同步。
5. **#101 是模板。** 所有人格參照 #101 的結構和品質標準鍛造。
6. **repos/ 不 push。** 太重，在 .gitignore 裡。
