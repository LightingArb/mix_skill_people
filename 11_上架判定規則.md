# 上架判定規則

> 鍛造完成後，決定產出是否進入 `鍛造間/` 的判定邏輯。
> 版本：3.0.0
> 最後更新：2026-03-31

---

## 前提

- 鍛造產出先存到各流派目錄：`鍛造間/codex/人格/`、`鍛造間/claudecode/人格/`、`鍛造間/codex/角色/`、`鍛造間/claudecode/角色/`
- 流派目錄內的檔案**帶流派前綴**（如 `codex_#308_xxx.md`）
- 上架區 `鍛造間/人格/` 和 `鍛造間/角色/` 內的檔案**無流派前綴**（通用版）
- 上架區是 SRI 唯一取用來源

---

## 四種判定情況

### 情況 1：架上沒有 ＋ 只有 1 個流派鍛造了

**判定：直接上架。**

```
流派 A 鍛造完成
  → 鍛造間/[流派A]/人格/[流派A]_#XXX_名稱.md
  → 檢查 鍛造間/人格/ 有無 #XXX
  → 沒有
  → 檢查其他流派有無 #XXX
  → 沒有
  → 去掉流派前綴 → 複製到 鍛造間/人格/#XXX_名稱.md
```

### 情況 2：架上沒有 ＋ 2 個流派都鍛造了

**判定：AI 讀兩份，融合後上架。**

```
兩個流派都有 #XXX
  → AI 讀 鍛造間/codex/人格/codex_#XXX_名稱.md
  → AI 讀 鍛造間/claudecode/人格/claudecode_#XXX_名稱.md
  → 融合規則：
      1. 結構以30_人格熔爐.md 的模板為準
      2. 內容取兩份的聯集（去重後）
      3. 衝突時保留更具體、更有證據的版本
      4. 素材追溯合併兩份的來源
  → 融合結果 → 鍛造間/人格/#XXX_名稱.md
```

### 情況 3：架上已有 ＋ 只有 1 個流派有新版

**判定：問使用者要不要覆蓋。**

```
流派 A 鍛造了 #XXX 新版
  → 鍛造間/人格/ 已有 #XXX
  → 只有流派 A 有新版
  → 問使用者：
      「架上已有 #XXX，流派 A 鍛造了新版。要覆蓋嗎？」
      A) 覆蓋 → 去掉流派前綴 → 覆蓋上架區
      B) 不覆蓋 → 新版只留在流派目錄
```

### 情況 4：架上已有 ＋ 2 個流派都有新版

**判定：問使用者要覆蓋還是融合後上架。**

```
兩個流派都有 #XXX 新版
  → 鍛造間/人格/ 已有 #XXX
  → 問使用者：
      「架上已有 #XXX，兩個流派都鍛造了新版。」
      A) 用流派 A 的版本覆蓋
      B) 用流派 B 的版本覆蓋
      C) 融合兩份後覆蓋（同情況 2 的融合規則）
      D) 不動，新版只留在流派目錄
```

---

## 融合規則細節

適用於情況 2 和情況 4-C。

### 結構融合

1. 章節順序以**30_人格熔爐.md 的模板**（或**31_角色熔爐.md 的模板**）為準
2. 不可增減章節
3. 兩份都有的章節 → 合併內容

### 內容融合

| 章節類型 | 融合方式 |
|---------|---------|
| 角色定位 | 取更完整的版本，補入另一版獨特的描述 |
| 核心原則 | 聯集去重。同一概念取更具體的版本。上限 8 條 |
| 思考框架 | 聯集去重。同名框架取步驟更完整的版本。上限 6 個 |
| 行為規則 | 聯集去重。衝突時保留更具體的 |
| 提問方式 | 聯集去重 |
| 審查維度 | 聯集去重 |
| 不做什麼 | 聯集（不去重，除非完全相同） |
| 素材追溯 | 合併兩份來源 |

### 衝突裁決優先序

1. 更具體的（有數字、有步驟）> 更抽象的
2. 有素材追溯的 > 無追溯的
3. 核心層來源 > 補強層來源
4. 無法判定 → 兩個都保留，標記 `[待人工裁決]`

---

## 上架後

- 上架完成 → 更新10_老闆.md 的架上清單（如有新增）
- SRI 取用時只看 `鍛造間/`，不看流派目錄

---

## 角色卡適用同樣邏輯

以上規則同時適用於人格（`鍛造間/人格/`）和角色（`鍛造間/角色/`）。把「人格」替換為「角色」即可。
