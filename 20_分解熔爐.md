## 系統定位

```
21_礦脈圖.md（素材來源清單）
  ↓ Step 0 比對
compiler.py + config.json
  ↓ 一致才繼續
本文件的編譯流程
  → output/
      ├── raw-cards/（碎片卡片）
      ├── 00_總索引.md（全卡片索引）
      ├── 01_分類彙整.md（六大分類歸類 ← 人格熔爐挑卡依據）
      ├── 02_工具清單.md
      └── compile_log.json
```

**01_分類彙整.md 是 compiler 自動產出的分類數據。人格熔爐鍛造時根據這份分類挑卡，不是自己分類。**

## 執行模式閘門

AI 執行任何操作前，必須先宣告當前模式：

| mode | 觸發指令 | 執行區段 | 產出位置 | 禁止碰 |
|------|---------|---------|---------|--------|
| `compile` | 「執行編譯」「重新編譯」 | 本文件全部 | output/ | 人格定義、角色定義 |

**compile 模式不得修改人格或角色。** 要改人格必須到30_人格熔爐.md，切 persona_edit 模式。

## Step 0：同步檢查（編譯前必做）

> **每次執行編譯前，AI 必須先跑這一步。不通過就不能繼續。**

### 0-1：讀取三方資料

1. 讀 **21_礦脈圖.md** 的 repo 清單，列出所有 enabled 的 repo name
2. 讀 **config.json** 的所有 entry，列出 enabled=true 的 repo name
3. 讀 **compiler.py** 頂部的 `REGISTERED_REPOS` 標記（見下方「compiler.py 同步標記規格」），列出已登記的 repo name

### 0-2：三方比對

```
礦脈圖 ∩ config ∩ compiler 全部一致？
  ├─ 是 → 輸出「✓ 三方同步，繼續編譯」→ 進入 Step 1
  └─ 否 → 停止，輸出不同步報告
```

### 0-3：不同步報告格式

```
⛔ 同步檢查未通過，編譯終止。

[不同步清單]
  礦脈圖有 / config 沒有：[repo 名稱]  ← 新增了 repo 但 config 沒跟上
  config 有 / 礦脈圖沒有：[repo 名稱]  ← config 有殘留
  config 有 / compiler 沒登記：[repo 名稱]  ← compiler 的 REGISTERED_REPOS 沒更新
  （其他差異...）

[下一步]
  請把以下三份檔案丟給 AI：
    1. 20_分解熔爐.md（本文件，內含更新規則）
    2. compiler.py
    3. config.json
  AI 會根據本文件「compiler / config 更新規則」區段，
  產出具體的修改方案，交給 Codex 執行。
  修改完成後重新執行編譯。
```

compiler.py 的檔案頂部必須維護一個 `REGISTERED_REPOS` 標記，讓 Step 0 可以機械式比對：

```python

# 本標記由分解熔爐 Step 0 比對用，手動維護。
# 格式：每行一個 repo name，與 config.json 的 name 欄位一致。
# 最後更新：2026-03-31
REGISTERED_REPOS = [
    "gstack",
    "superpowers",
    "oh-my-claudecode",
    "get-shit-done",

    "pm-skills",
    "marketingskills",

    "awesome-startup",
    "web-access",
    "gcloud-mcp",
    "chrome-devtools-mcp",

# === END REGISTERED_REPOS ===
```

AI 更新 compiler.py 時必須同步更新此標記。

## compiler / config 更新規則

> 當 Step 0 不通過時，AI 讀這段來產出修改方案。

### 更新 config.json

**新增 repo：**

在 config.json 加一個 entry，按以下規則判斷欄位：

| 欄位 | 判斷規則 |
|------|---------|
| name | repo 簡稱（用於資料夾名，不含 owner） |
| url | GitHub clone URL |
| type | 有 SKILL.md → `skill`；README 知識型 → `knowledge`；只是工具 → `tool` |
| scan_targets | 根據 repo 結構：有 SKILL.md → `["SKILL.md"]`；有 CLAUDE.md → 加入；有 AGENTS.md → 加入 |
| scan_root_readme | skill 通常 false，knowledge 通常 true |
| max_readme_depth | 預設 1 |
| enabled | true |
| exclude_patterns | 看 repo 有無非相關的大目錄需排除 |
| extract_policy | 按 repo 特性設定（見21_礦脈圖.md 的欄位定義） |
| note | 從礦脈圖的備註複製 |

**移除 repo：** enabled 改 false，不刪 entry，加 note 標記原因和日期。

**替換 repo：** 舊的 enabled=false + 新的加 entry。

### 更新 compiler.py

**需要改的情況：**

| 情況 | 改什麼 |
|------|--------|
| 任何 repo 新增或移除 | 更新 `REGISTERED_REPOS` 標記 |
| 新 repo 需要精確的一句話定位（gstack/superpowers 等級） | 加 `ONE_LINERS` 條目 |
| 新 repo 的自動分類不準 | 加 `CATEGORY_OVERRIDES` |
| 新 repo 的自動領域推斷不準 | 加 `DOMAIN_OVERRIDES` |
| 新 repo 的 scan_targets 格式不同（如 `categories/**/*.md`） | 修改 `iter_target_files()` |

**不需要改的情況：** 新 repo 是標準 SKILL.md 結構 + 自動推斷品質可接受 → 只更新 `REGISTERED_REPOS`，不動其他程式碼。

**判斷方式：** AI 先 clone 新 repo，快速看 3-5 個 SKILL.md 結構，跟 gstack/superpowers 比較，再決定要不要加手動覆蓋。

### 更新順序

固定順序，不可跳步：
1. 更新 config.json
2. 更新 compiler.py（至少更新 `REGISTERED_REPOS`）
3. 重新執行 Step 0 確認三方同步
4. 才能進入 Step 1 編譯

## Step 1-8：編譯流程

> 使用者說「執行編譯」，Step 0 通過後，AI 按以下步驟執行。

### Step 1：Clone

- 讀 config.json
- 對每個 enabled=true 且 type≠tool 的 repo：已存在 → git pull；不存在 → git clone
- 失敗的 repo 記錄錯誤，繼續

### Step 2：掃描

- 按本文件「掃描規則」找出要處理的 .md 檔案
- 套用 exclude_patterns 和 extract_policy

### Step 3：拆卡

- 按本文件「拆卡規則」產出 raw-card
- 深度抽取：完整讀取每個 SKILL.md，不截斷
- 寫入 output/raw-cards/[repo]/[卡片].md

### Step 4：去重

- 按 canonical_group 分組，同組只保留 language=en 或最長版本
- 記錄去重數量

### Step 5：品質檢查

- 按本文件「品質規則」的 5 項檢查逐條驗證
- 檢查停止條件

### Step 6：產出索引

- output/00_總索引.md（主索引 + 附錄 + 統計 + 關聯矩陣）
- output/01_分類彙整.md（六大分類）
- output/02_工具清單.md
- output/compile_log.json

### Step 7：報告

- 輸出編譯報告，等待使用者確認

### Step 8：確認後更新

| 級別 | 條件 | 可更新 |
|------|------|--------|
| 普通編譯 | 使用者確認 OK | 版本時間、素材快照 |
| 人格重鍛 | 使用者明確說「重鍛 #10X」 | 版本時間、指定人格、素材快照 |

**普通編譯不得動人格定義。** 要改人格必須到30_人格熔爐.md 操作。

## 掃描規則

> compiler.py 的 `iter_target_files()` 實作這些規則。

### 掃描目標

- type=skill：掃描所有符合 scan_targets 的檔案（遞迴搜尋）
- type=skill 且 scan_root_readme=true：也掃描 depth ≤ max_readme_depth 的 README.md
- type=knowledge：只掃描根目錄的 README.md，按 `##` 標題拆 section
- type=tool：不掃描，只記錄到 02_工具清單.md
- type=reference 或 enabled=false：完全跳過

### 排除規則

- 排除 config.json 中 exclude_patterns 匹配的路徑（glob 格式）
- 排除 extract_policy.skip_patterns 匹配的路徑
- 排除 node_modules/、.git/、__pycache__/
- 排除非 .md 檔案
- 路徑含多語目錄（ja-jp/、ko-kr/、pt-br/、zh-cn/、zh-tw/）→ 標記為 translation，產卡但不進主索引

## 拆卡規則

> compiler.py 的 `extract_skill_card()` 和 `build_card_markdown()` 實作這些規則。

### skill 類型卡片必須包含的欄位

| 區塊 | 內容 | 抽取方式 |
|------|------|---------|
| 來源 | repo、路徑、檔案類型、card_kind、language、canonical_group | 自動判定 |
| 一句話定位 | 精確描述 | gstack+superpowers 用 ONE_LINERS 對照表；其他用 description 第一句 |
| 核心人格特質 | 逗號分隔 | infer_traits() |
| 核心思考框架 | 從 heading keywords 深挖，上限 18 行 | collect_framework_lines() |
| 核心行為規則 | must（12 行）+ forbid（12 行） | collect_behavior_lines() |
| 提問方式 | 問句 + AskUserQuestion | collect_question_lines() |
| 審查維度 | 僅審查型 | collect_review_lines() |
| 輸出格式要求 | format/output heading | collect_format_lines() |
| 適用場景 | 觸發條件推斷 | build_applicable_lines() |
| 原文精華摘錄 | 5-20 句原文 | collect_quote_lines() |
| 關聯 | 跨卡片分析 | compute_relations() |
| 分類標記 | 六大類 checkbox | infer_categories() |

### card_kind 判定順序

1. 路徑含多語目錄 → `translation`
2. 檔名含 template → `template`
3. 檔案是 README.md → `overview`
4. 檔案是 AGENTS.md → `agent`（若內含 ≥2 個行為信號）或 `overview`
5. 檔案是 CLAUDE.md → `overview`
6. 路徑含 installer/setup/upgrade/configure/hooks → `tool-reference`
7. 檔案是 SKILL.md → `skill`
8. 其他 → `overview`

### 一句話定位規則

| 來源 | 處理 |
|------|------|
| gstack + superpowers | 使用 ONE_LINERS 手動對照表（48 條，見 compiler.py） |
| 其他 | frontmatter description 第一句；< 10 字則用 fallback_description() |
| 禁止 | 「專注於 X 與 Y 任務的 skill，適合處理 Z 相關工作」套版句型（0 容忍） |

### 六大分類

```
思考框架型 / 審查型 / 工具程序型 / 人格型 / 流程型 / 元技能型
```

gstack + superpowers 使用 CATEGORY_OVERRIDES（見 compiler.py）。其他 repo 用關鍵字推斷。

### knowledge 類型卡片（簡化格式）

只含：來源、卡片元資料、主題、核心知識、適用場景、分類標記。
不做深度抽取（無 framework/behavior/question）。

## 品質規則

### 編譯後自動檢查（5 項）

| # | 檢查項 | 不通過處理 |
|---|--------|-----------|
| 1 | 一句話定位無套版句型 | 列入 warnings，標記卡片 |
| 2 | gstack + superpowers 全部使用 ONE_LINERS | 編譯失敗 |
| 3 | card_kind 正確（README ≠ skill） | 列入 warnings |
| 4 | translation 已隔離到附錄 | 列入 warnings |
| 5 | 每張 skill 至少有 framework 或 behavior 其一有內容 | 列入 quality_checks |

### compile 模式停止條件

以下情況 AI 必須停止，不得自動往下做：

| 條件 | 處理 |
|------|------|
| Step 0 三方不同步 | 停止，輸出不同步報告 |
| warnings 超過 5 條 | 只更新版本時間和報告，不更新素材快照 |
| 主索引卡片數變動超過 ±15% | 先人工複核 |
| agent 仍為 0 | 禁止重鍛 #203 |
| empty_framework 或 empty_behavior 比上一版更差 | 禁止直接更新人格引用 |
| 新 repo 替換舊 repo 但無法覆蓋原核心素材 | 禁止直接替換人格引用 |

## 素材變動流程

### 場景 A：新增 repo

1. 在21_礦脈圖.md 加一行（異動記錄 + repo 資訊）
2. 執行編譯 → Step 0 會攔住（不同步）
3. 按輸出指引同步 config.json + compiler.py
4. 重新執行編譯 → Step 0 通過 → 繼續
5. 確認新 raw-cards 品質
6. 判斷：新素材改變現有人格？
   - 是 → 到30_人格熔爐.md 重鍛受影響的人格
   - 否 → 只更新素材快照
7. push

### 場景 B：移除 repo

1. 在21_礦脈圖.md 標記移除
2. 執行編譯 → Step 0 攔住
3. 同步 config.json（enabled=false）+ compiler.py（更新 REGISTERED_REPOS）
4. 重新執行編譯
5. 確認：被移除 repo 有無卡片被現有人格引用？
   - 有 → 到30_人格熔爐.md 重鍛
   - 沒有 → 只更新素材快照
6. 上架區已鍛造的人格不會自動消失
7. push

### 場景 C：替換 repo

1. 場景 B（移除舊的）+ 場景 A（新增新的）
2. 特別確認：新 repo 能否完全替代舊的核心素材？
3. 更新順序固定：先改礦脈圖 → 再同步 config + compiler → 再跑編譯

## ONE_LINERS 與 OVERRIDES 參考

gstack + superpowers 的 48 條 ONE_LINERS 和 CATEGORY_OVERRIDES、DOMAIN_OVERRIDES 完整資料在 compiler.py 中。

覆蓋原則：
- gstack + superpowers 的每張卡片都有手動標記的分類和領域
- 手動標記優先於自動推斷
- 其他 repo 不做手動覆蓋，使用自動推斷
